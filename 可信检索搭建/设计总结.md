# 可信检索搭建 Skill（元层次）— 设计总结

> 撰写日期：2026-07-24
> 对应 Skill 版本：v2.2
> 文档定位：向企业导师/同事说明「为什么这样设计」的完整思路

---

## 一、一句话概述

一个让 Agent **在陌生数据领域现场搭建可靠检索方案**的元层次 Skill——不预装任何信源知识，而是教 Agent「如何发现信源、如何评估可靠性、如何构建白名单」。是人口/销售两个预置 Skill 背后那套六模块方法论的自我复制版本。

---

## 二、设计理念

### 理念 1：元层次——从「做菜」到「写做菜的菜谱」

人口数据 Skill 和销售数据 Skill 是两道菜。六模块是工序。元层次 Skill 是把工序本身写成菜谱——给 Agent 的不是「怎么查人口」，而是「怎么为一个新领域设计检索方案」。

正式定义：元层次（meta-level）= 关于 X 本身的 X。它不直接解决用户问题，而是让 Agent 获得「为新问题搭建解决方案」的能力。设计野心：**不是多做一个 Skill，是让这个 Skill 系列从此可以自我复制。**

### 理念 2：白名单是产出物，不是输入物

人口/销售 Skill 的信源白名单是预装的（写死在 SKILL.md 里）。元层次 Skill 没有预装的白名单——白名单是 Agent 现场构建的产出物。

这个设计破除了一个伪问题：既然没有白名单，Agent 要不要「破例」搜非白名单网站？答案是——没有白名单就没有「破例」。阶段一的任务就是构建白名单，构建完成之前不存在禁区。

### 理念 3：先定义失败（继承自人口数据）

```
失败模式                           →  反向推出的设计决策
─────────────────────────────────────────────────────────
错把领域 A 的信源结构套到领域 B     →  强制领域分析先行（先问「有没有唯一权威」）
不知道去哪找可靠信源                →  三途径发现（记忆 + 搜引用 + 实地检查）
SEO 内容农场污染搜索结果            →  搜引用不搜推荐 + 英文辅助搜索
候选太多、评估成本爆炸              →  两阶段过滤 + 达标即停
信源评估标准不统一                  →  可执行的检查项：身份/溯源/方法论/时效/机构
归因当数据                          →  数据/归因信源分离
补搜无限循环                        →  总预算 ≤2 轮 + 补位必先深评
```

### 理念 4：约束大于判断（继承）

不给 Agent 需要主观判断的指令，给可机械执行的硬规则。最典型的例子：信源评估拆成五个可执行的检查动作（身份/溯源/方法论/时效/机构），每个都有明确的通过/不通过标准，而非「判断这个网站可不可靠」。

### 理念 5：搜引用不搜推荐（元层次独有）

人口/销售 Skill 用预装白名单绕过了 SEO 问题。元层次没有白名单，必须在开放网页中发现信源。解决方案：不搜「什么是可靠信源」（SEO 重灾区），搜「谁在使用什么信源」——看这个领域的从业者、研究者在引用什么，不看谁在说自己好。配合英文搜索绕过中文内容农场。

## 三、架构

### 七步流程（六模块 + 领域分析前置）

```
第零步：检查现有 Skill（已有预置 → 转交，无 → 继续）
     ↓
阶段一：领域分析
  第一步：判断信源结构（树状 / 网状 / 单源垄断）
  第二步：发现候选信源（记忆 + 搜引用 + 实地检查）
  第三步：两阶段评估（低成本过滤 → 排序 → 达标即停）
  第四步：构建白名单
  第五步：显式记录（含槽位终态、储备清单）
  第六步：落盘（领域档案）
     ↓
阶段二：执行标准检索
  搜索规划 → 信源约束 → 接地规则 → 补搜 → 自检 → 可审计交付
```

### 跟前两个 Skill 的架构差异

| 模块 | 人口/销售 Skill | 元层次 Skill |
|------|----------------|-------------|
| 领域分析 | 不需要（领域已知） | **新增前置于六模块之前**——白名单的生成器 |
| 信源约束 | 预装白名单 | 动态构建的白名单 |
| 补搜机制 | 领域特定触发器 | 通用触发 + 储备清单补位（补位必先深评） |

领域分析模块是这个 Skill 的真正创新点——它把「数据工程师为一个新领域设计检索方案」这个人工工序自动化了。

### 封槽机制：达标即停（下限封槽）

核心设计：第二遍深评不是评完所有候选，而是按排序逐个评，每个槽位达到下限即封槽。

| 结构 | 槽位 | 封槽条件 |
|------|------|---------|
| 树状 | 每优先级层级 | 该层 ≥1 即封 |
| 网状 | 数据信源 | ≥2 独立平台即封 |
| 网状 | 归因·法定披露 | ≥1 即封 |
| 网状 | 归因·第三方审计 | ≥1 即封 |
| 网状 | 归因·第三方观点 | ≥2 即封 |
| 单源垄断 | 唯一信源 | ≥1 即封 |

每个槽位终态只能是三者之一：已填 / 空槽标注 / 不适用。**沉默跳过即违规。** 这个规则是场景 2 实测翻车后补入的——网状归因三槽位原本嵌套在一行里，subagent 跳过两个，翻车后拆成独立行。

## 四、关键设计决策

### 决策 1：为什么领域分析放在六模块前面，而不是在信源约束里做？

因为信源约束模块原本假设白名单是已知的。元层次的白名单是未知的——需要在碰任何数据之前先搞清楚「这个领域长什么样」。领域分析不是信源约束的子步骤，而是它的前置生成器。

### 决策 2：为什么选「达标即停」而不是「评估所有候选」？

成本控制。最坏情况：15 个候选，每个深评需要 3 个页面打开 = 45 次工具调用——还没开始搜数据。达标即停把成本从「候选数 × 深评成本」压到「槽位数 × 深评成本」，与领域复杂度脱钩。

代价是白名单质量被排序信号锁死——如果第一遍的弱信号把真正的权威信源排到了后面，早退会用次优信源填槽。实测中这个风险由独立性判定兜底（两个独立信源的平均能压住个别次优）。

### 决策 3：为什么归因信源拆三个子槽位？

销售数据 Skill 的经验——归因来源本身有可信度差异：公司年报（法定披露，有监管约束）≠ IOSCO 审计（第三方专业鉴证）≠ 券商研报（独立观点，需 ≥2 家交叉）。不能归为一类。元层次把这个分层结构化了。

### 决策 4：为什么加「储备清单」？

达标即停必然遗留未评候选。与其丢弃或用「不合适」模糊标注，不如保留为储备——阶段二补搜时优先从储备补位（补做深评后才可用），而非重新发现信源。未评候选从「现在必须评的负债」变成「将来按需取用的资产」。

### 决策 5：为什么强调「英文辅助搜索」？

中文搜索结果中内容农场、SEO 优化文章占比高。英文搜索结果的学术/机构占比通常更高。这不是理论判断——是人口/销售 Skill 开发过程中的实操经验：用中文搜「人口数据」第一页多是百度百科和自媒体，用英文搜「population data」第一页是 World Bank 和 UN。

## 六、优越性

### 1. 自我复制——从两个 Skill 到一个 Skill 系列

元层次 Skill 将六模块方法论从人工操作升级为 Agent 可执行的元能力。它的自然产出物（领域档案）经过人工审核即可升级为新预置 Skill——人口/销售就是这么诞生的，下一个不再需要从头设计。

### 2. 领域无关——一套方法论覆盖三种结构

树状（有唯一权威）、网状（多源无权威）、单源垄断——三种信源结构的发现和评估策略全部内建。实测六个场景覆盖全部三种结构。

### 3. 成本可控——达标即停

深评成本不随候选丰度增长。六场景实测中，阶段一与阶段二的工具调用比稳定在 4:6 左右，未出现「建白名单比搜数据还久」的情况。

### 4. 可审计——每个决策可追溯

领域分析记录包含完整的探索过程、淘汰原因、槽位终态、排序依据、储备清单。不仅是「搜到了什么」，还包含「为什么信这个」、「淘汰了什么」、「还有什么备用」。实测翻车（场景 2 归因子槽位缺失）正是因为可审计性——沉默跳过能被发现。

### 5. 实战验证——不是纸面设计

六场景实测 + 一个真实翻车 + 修复复测闭环。跟人口/销售 Skill 同款质量流程。STATUS.md 诚实标注开发状态，不掩盖未测项。

## 八、已知局限（诚实标注）

1. **排序信号质量未被系统验证**：达标即停的质量依赖于第一遍信号能正确排序。实测中排序偏保守但不致命（储备清单质量好、独立性判定兜底）。权威信源低调、内容农场张扬的极端场景尚未触发。
2. **深评结论可复现性**：同一信源在不同轮次运行中可能得到不同评估结论（如百川盈孚在场景 4 通过深评、场景 2 复测中进储备清单）。独立运行可解释，但尚未建立跨轮一致性标准。
3. **非中英双语领域未测**：日语、葡语等非中英信源的关键词翻车风险未覆盖。
4. **领域档案跨会话复用未测**：第二次同领域查询是否真的读取并复用了档案。
5. **单源垄断的豁免边界**：什么算「真·单源」、什么算「找不到第二源」，边界由 Agent 判断，存在误判风险。
