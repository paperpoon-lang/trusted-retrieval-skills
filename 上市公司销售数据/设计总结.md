# 上市公司销售数据检索 Skill — 设计总结

> 撰写日期：2026-07-20
> 对应 Skill 版本：v1
> 文档定位：向企业导师/同事说明这个 Skill「为什么这样设计」的完整思路

---

## 一、一句话概述

一个让 Agent **可靠地从法定披露平台检索上市公司营收数据**的 Skill——与人口数据 Skill 共享六模块骨架，在信源结构上做了领域适配：数据/归因分离、审计状态分层、多触发器补搜。

---

## 二、设计理念

### 理念 1：窄领域优势（继承自人口数据）

两个独立 Skill 而非一个通用数据 Skill——根本原因不变：覆盖领域越窄，信源清单越精确可执行。人口数据用的是各国统计局，销售数据用的是各市场法定披露平台——不可混用。

### 理念 2：数据/归因分离（销售数据独有）

人口数据不存在「观点当数据」的风险——百度百科和维基百科直接禁用就行。但销售数据领域，券商研报是**有价值但有风险**的信源：它的营收预测不能当实际数据用，但对差异的分析值得引用。

因此本 Skill 在信源约束层做了人口数据没有的结构性拆分：

```
数据信源（只允许法定披露）
├── A 股：巨潮资讯网 + 上交所/深交所/北交所
├── 港股：港交所披露易
└── 美股：SEC EDGAR（10-K/20-F/10-Q/8-K）

归因信源（只解释差异，不提供数字）
├── [法定披露]：公司 MD&A、问询函回复
├── [第三方·审计]：独立审计报告
└── [第三方观点]：券商研报（≥2 家建议）
```

### 理念 3：先定义失败（继承自人口数据，扩展至销售特有场景）

```
失败模式                            →  反向推出的设计决策
─────────────────────────────────────────────────────
Agent 拿券商研报预测数当营收用      →  数据/归因分离
Agent 把业绩快报当最终数据交付      →  审计状态分层 + 未定稿触发
Agent 用了更正前的旧数字            →  重述触发（优先级 #1）
Agent 忽略保留意见直接报数          →  审计非标触发 + ⚠️ 显式标注
Agent 把 Q1+Q2 加总伪称直接披露     →  派生值规则：[计算值] 标注
Agent 自行推理两地年报差异原因      →  准则标注（事实，允许）vs 原因解释（须引用，禁推理）
```

## 三、六模块架构（与人口数据对比）

两个 Skill 共用同一套六模块骨架，每个模块只做领域替换：

| 模块 | 人口数据 | 上市公司销售数据 | 变化性质 |
|------|---------|----------------|---------|
| **信源约束** | 6 级官方信源（国家统计局→学术） | 数据信源 ⊕ 归因信源分离 + 审计状态分层 | 结构性变化 |
| **搜索规划** | 国家/指标/时间/粒度 + 本地化关键词 | 公司/市场/指标/财年口径/币种 + 字段名对照表 | 维度替换 |
| **接地规则** | 数字只来自本轮检索 | 同上 + 差异原因只引用归因信源 + 派生值规则 | 增强 |
| **补搜机制** | 门控 2 轮 + 学术库强制触发 | 5 触发器（重述/未定稿/审计非标/冲突/缺失） | 触发器替换 |
| **自检反思** | 五维检查 | 五维 + 第六维差异归因溯源 | 增强 |
| **可审计交付** | 结构化输出 + 溯源 URL | 同上 + 三档标注 + 准则事实标注 vs 归因引用分离 | 增强 |

## 四、关键设计决策

### 决策 1：为什么 A 股首选巨潮资讯网？

巨潮是证监会指定的统一信息披露入口，覆盖沪深京三市。上交所/深交所/北交所各自官网也是法定披露渠道（同等权威），但只覆盖自己市场的公司。巨潮作为「集中入口」而非「更权威」的理由：搜一家公司不用先判断它上的是沪市还是深市。

### 决策 2：为什么审计状态要分四层？

同一家公司同一时期可能存在多个层级的数据——业绩快报先出、年报后出、更正公告再修正。Agent 必须知道哪个更可信：

| 层级 | 标注 | 设计理由 |
|------|------|---------|
| 已审计 | 无需额外标注 | 经独立审计，最高可信 |
| 未经审计 | `[未经审计]` | 季报/半年报，未走完审计程序 |
| 初步/预估 | `[初步数据·可能修正]` | 快报/预告/8-K Ex-99，数字可能后续调整 |
| 历史参考 | `[招股书数据]` | 非当期数据 |

### 决策 3：为什么归因信源分三档？

人口数据的冲突处理只需「同时呈现 + 不选边」。销售数据的归因来源本身有可信度差异：

- 公司 MD&A → 有监管约束但非独立 → `[法定披露]`
- 独立审计报告 → 第三方专业鉴证 → `[第三方·审计]`
- 券商研报 → 未经独立验证 → `[第三方观点]`

不能把公司自述和审计报告放同一档，也不能把审计报告和券商研报放同一档。

### 决策 4：为什么港股关键词要配繁体中文？

披露易平台的文件标题多数为繁体中文（如「截至2024年12月31日止年度的經審計業績公告」），英文关键词无法命中。这是场景 1 实测暴露的 bug，修复方案与人口数据中「日本用『人口推計』」同构。

### 决策 5：为什么派生值必须标注 `[计算值]`？

A 股季报采用累计口径，Q2 单季营收必须通过 H1 − Q1 计算获得。如果不强制标注，Agent 可能把计算值伪装成直接披露数字。规则核心：**未标注的计算值禁止输出**。

## 六、优越性

### 1. 架构可迁移——复用同一个骨架

从人口数据到销售数据，六模块骨架不需要改动，只需要替换每个模块的领域内容。这验证了系列设计的可迁移性。

### 2. 数据/归因分离——防住了人口数据不存在的风险

券商研报的营收预测不能当实际营收——这条规则在人口数据里不适用（没有「百度百科的人口预测」当数据用的场景），但在销售数据里是核心防御。

### 3. 审计状态分层——把「最新」和「最可信」区分开

业绩快报的数字比年报新，但年报更可信。分层机制确保 Agent 不会因为追求时效性而牺牲可靠性。

### 4. 五触发器补搜——比人口的单一学术触发更完整

人口数据的补搜主要靠学术库强制触发（TFR 等高敏感指标）。销售数据的五个触发器覆盖了更广泛的失效模式：重述、未定稿、审计非标、冲突、缺失。

### 5. 可审计——每个数字可溯源、每条差异有引用

三档标注 + 准则事实标注 vs 归因引用分离，确保交付物不仅「有数据」，而且「能看到数据从哪来、解释从哪来」。

## 七、测试覆盖

六个实测场景，覆盖全部 10 条新机制：

| # | 场景 | 核心验证 | 结果 |
|---|------|---------|:---:|
| 1 | 中国平安 A+H 冲突归因 | 繁体关键词修复 + 准则标注 + 不选边 | ✅ |
| 2 | 阿里 20-F 财年口径 | 20-F 命中 + 财年 ≠ 自然年 + Revenue 映射 | ✅ |
| 3 | *ST 观典 业绩快报未定稿 | `[初步数据·可能修正]` + 补搜 + 法定截止 | ✅ |
| 4 | 佳合科技 小盘归因 | 观点 ≠ 数据 + `[第三方观点·单一来源]` | ✅ |
| 5 | *ST 星农 更正+审计非标 | 重述触发 #1 + 审计非标触发 #3 | ✅ |
| 6 | 茅台 Q2 单季派生值 | `[计算值]` 标注 + 原始披露溯源 | ✅ |

> 已知未测项：无归因负面分支（「未找到公开归因分析」标注）和汇率换算变体。

## 八、已知局限（诚实标注）

1. **小盘/冷门标的归因覆盖不足**：零券商覆盖时只能标注「未找到公开归因分析」——这条标注已设计但实测中未触发（候选标的恰好有 1 家券商）。
2. **汇率换算未测**：派生值规则的汇率变体（如人民币→美元换算）尚未实测。
3. **URL 可访问性**：Agent 无法预判 SEC/披露易链接是否被墙或需要特殊网络环境。
