# 人口数据检索 Skill — 设计总结

> 撰写日期：2026-07-13
> 对应 Skill 版本：v1.2
> 文档定位：向企业导师/同事说明这个 Skill「为什么这样设计」的完整思路

---

## 一、一句话概述

一个让 Agent **可靠地从官方一手信源检索人口数据**的 Skill——核心不是「能搜到数据」，而是「搜到的数据可信、可追溯、可审计」。

---

## 二、设计理念

### 理念 1：先定义失败，再设计成功

我们没有从「这个 Skill 应该做什么」开始，而是先问「它在什么情况下算废了」。

```
失败模式          →  反向推出的设计决策
─────────────────────────────────────
搜错了信源       →  信源约束（白名单 + 优先级）
搜到了但没认出   →  接地规则（只用检索到的数据）
没搜到就放弃了   →  补搜机制（门控补充检索）
搜出来了但是假的 →  自检反思 + 可审计（溯源 + 冲突处理）
```

**好处**：每个失败模式都逼出一条具体、可测的设计决策。验收标准天然可测——「不能从百度百科拿人口数据」比「应该用权威数据」好测 100 倍。

### 理念 2：约束大于判断

不给 Agent 需要主观判断的指令，给可机械执行的硬规则。

- ❌ 「当官方数据存在争议时，补搜学术数据库」→ Agent 判断不了什么叫「争议」
- ✅ 「当指标为 TFR / 预期寿命 / 婴儿死亡率时，无论首轮结果如何，强制补搜学术数据库」

### 理念 3：窄领域优势

Skill 覆盖领域越窄，越能写出可执行的必看信源清单——把最大风险点（信源选择）从 Agent 手里拿回来。这是我们做「人口数据」和「上市公司销售数据」两个独立 Skill、而非一个通用数据 Skill 的根本原因。

### 理念 4：接地优先于引用

「只引用本轮实际检索到的内容」比「事后标注来源」更重要。事后标注不能防止模型编造——接地机制（检索结果直接进上下文，模型没有「我记得某数据」的自由）才能。

## 三、六模块架构

```
        ┌─────────────────────────────┐
        │   搜索规划 ←──→ 信源约束      │  ← 地基层（互锁）
        └─────────────┬───────────────┘
                      ↓
        ┌─────────────────────────────┐
        │         接地规则              │  ← 执行层
        └─────────────┬───────────────┘
                      ↓
        ┌─────────────────────────────┐
        │         补搜机制              │  ← 兜底层
        └─────────────┬───────────────┘
                      ↓
              ┌──────────────┐
              │   自检反思    │←──── 缺口 → 回到搜索规划
              └──────┬───────┘
                     ↓ 通过
              ┌──────────────┐
              │   可审计交付  │  ← 交付层
              └──────────────┘
```

| 模块 | 防止的失败 | 核心机制 |
|------|-----------|---------|
| **信源约束** | 搜错信源 | 6 级信源分级 + 白名单 + 独立性判定 + 豁免规则 |
| **搜索规划** | 不知道搜什么 | 需求四维分析（国家/指标/时间/粒度）+ 国家→统计局映射 + 本地化关键词 |
| **接地规则** | 凭记忆胡说 | 只回答本轮检索到的数据；官方未公布≠失败 |
| **补搜机制** | 一次搜不到就放弃 | 门控补充检索（最多 2 轮）+ 学术库强制触发 |
| **自检反思** | 交付了错的 | 五维检查（完整性/可追溯/时效性/一致性/边界），未过回补搜 |
| **可审计交付** | 用户无法验证 | 结构化输出 + 溯源 URL + 冲突处理（不选边站） |

### 架构的三个关键判断

1. **信源约束和搜索规划是「互锁地基」，不是上下级。** 「搜什么」和「在哪搜」是互相约束、并行决策的——搜索目标影响信源选择，信源能力也反过来影响搜索策略。

2. **信源约束是「横切关注点」（cross-cutting concern）。** 它逻辑上是搜索规划的子集，但因为影响全局、是最大风险点，被提升为独立模块。类似软件工程中的安全、日志。

3. **自检反思是回路，不是终点。** 自检发现缺口 → 回到搜索规划补搜，形成闭环。这本质上是一个有界的 ReAct 循环。

## 四、借鉴了什么

这个 Skill 不是闭门造车，而是博采众长。每一处借鉴都做了「借鉴 + 取舍 + 解释」：

| 来源 | 借鉴了什么 | 如何取舍 |
|------|-----------|---------|
| **Pickaxe Grounding Stack** | 三层验证架构（实时信源→检索接地→引用规则）；「模型永不凭记忆回答」原则；六条引用规则 | 借鉴为「接地规则」独立模块 + 可审计交付 |
| **OpenAI GPT-5.6 Prompting Guide** | 精简优先（删除比添加有效）；结果导向不写脚本；约束代替判断；完成后自我验证 | 用于指令写法：v1.2 做了骨架精简；学术触发用「约束代替判断」 |
| **Agentic RAG 企业指南** | Plan→Retrieve→Act→Reflect→Cite 核心循环 | Reflect 步骤 → 我们的「自检反思」模块 |
| **Anthropic Agent Skills Spec** | SKILL.md 文件格式；三级渐进加载；description 即触发 | 直接遵循开放标准，保证跨工具兼容 |
| **Anthropic skill-creator** | 写 Skill 的方法论流程（意图→草稿→测试→迭代） | 我们的完整工作流：设计→编写→四测试→三轮迭代 |
| **沙利文 Agent 的 GSR** | 门控补充检索（ReAct 工程化变体）：搜索→审查→补搜，有界循环 | 直接复用为「补搜机制」，但解绑了沙利文特定管线 |
| **学术信息素养（三角验证）** | 至少三个独立可信来源交叉验证；单一来源不可靠 | 借鉴为「至少 2 个独立信源」+ 独立性判定 |

## 五、优越性

相比「让 Agent 自由搜索人口数据」的朴素做法，这个 Skill 的优越性：

### 1. 可信性——从「能搜到」到「搜到的是对的」

- 白名单机制杜绝了百度百科、维基百科等非权威信源
- 接地规则杜绝了 Agent 凭训练数据「编」数据
- 学术库强制触发机制，对高敏感指标（如 TFR）做区间验证

### 2. 可审计性——每个数字可溯源

- 每个数据点都带来源机构、年份、URL、检索时间
- 交付格式标准化，冲突时同时呈现、不替用户做判断

### 3. 鲁棒性——边界场景有预案

实测四个场景全部正确处理：
- 基础查询（中国总人口）→ 双源验证
- 国际查询（日本）→ 本地化关键词命中官方
- 数据缺失（中国 TFR）→ 官方未公布 + 学术估计替代
- 数据冲突（布隆迪）→ 普查 vs 国际估计，标注差异不选边

### 4. 可扩展性——架构可迁移

六模块架构不绑定人口数据。上市公司销售数据 Skill 可复用同一骨架，只需替换信源清单和检索策略。兼容版 Skill 则将这套方法上升到元层次。

### 5. 可维护性——方法论与规则解耦

信源清单、触发规则、输出格式都是显式的文档条目，非硬编码。调整判断标准只需改文档，符合「方法论与代码解耦」原则。

## 六、迭代历程（体现工程严谨性）

| 版本 | 变化 | 驱动 |
|------|------|------|
| v1.0 | 六模块初版（仅覆盖中国） | 设计定稿 |
| v1.1 | 支持全球多国 + 7 处修复（子站判定、二次报道标注、官方不公布处理、冲突话术、媒体白名单、豁免链接、学术强制触发） | 四个测试发现问题 + 两轮同行评议 |
| v1.2 | 结构精简（信源规则合并、交付模板清单化） | 骨架优化评议 |

**关键方法**：每一轮迭代都遵循「先测试，再改一条，验证后再改下一条」——不一次全重写。这来自 OpenAI 的 prompt migration workflow：「Do not rewrite a working prompt stack all at once.」

## 七、已知局限（诚实标注）

1. **二次报道依赖**：部分国家官方站点难以访问时，依赖国家级通讯社转载，标注 `[引用]` 但可靠性低于直接官方页。
2. **URL 可访问性**：Agent 无法预判链接是否被墙或失效。
3. **同源判定靠启发式**：域名/机构名比对是启发式，极端情况可能误判。

这些局限均已在 Skill 或测试报告中显式记录，不掩盖。
